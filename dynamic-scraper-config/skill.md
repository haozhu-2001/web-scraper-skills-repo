# Dynamic Scraper Config

配置动态抓取（Playwright 浏览器）的抓取规则。

## 何时使用

- 用户需要为某个需要浏览器渲染的网站配置抓取字段
- 用户需要定义抓取的 URL 模式和输出格式
- 用户需要调整已有抓取规则的字段映射

---

# 浏览器任务指令：列表页 XPath 解析

**核心原则：先源码，后验证。源码分析完成之前，不允许在真实页面上执行任何 XPath。**

---

## 第一步：获取页面源码（强制，不可跳过）

通过以下方式获取页面原始 HTML 源码：

- 使用 `view-source:` 协议打开页面源码
- 或直接 fetch 页面的原始 HTML 文本

**为什么必须用源码而不是页面 DOM？**

- 源码是服务端返回的静态 HTML，结构完整、层次清晰，没有 JS 动态插入的干扰
- 在源码文本中搜索、定位、数数量，比在动态 DOM 中摸索准确得多
- 页面 DOM 只在最后验证阶段使用，用于确认源码得出的 XPath 在实际渲染后仍然有效

---

## 第二步：在源码中完成全部结构分析

**注意：以下所有操作都在源码文本中进行。不要打开开发者工具的元素面板，不要检查实时 DOM，只用源码。**

### 2.1 在源码中定位一个条目

从页面正文区域（肉眼看到的列表）中随便记下一个条目标题的文字内容。在源码文本中搜索这段文字，找到它在 HTML 中的精确行位置。

### 2.2 在源码中回溯找容器

从该位置向上逐行回溯，找到包裹该条目的最外层容器元素。然后检查它的相邻兄弟：

- 兄弟元素标签名一致吗？
- 兄弟元素内部的子节点层级一致吗？
- 每个兄弟容器内都能找到一个带标题文字、带 `href` 的 `<a>` 标签吗？
- 这组容器在源码中的位置，是在页面主体区域而不是 `<nav>`、`<aside>`、`<footer>` 或横向滚动区吗？

全部符合 → 这就是容器锚点。**在源码中数一下这组容器一共重复了多少次**，记住这个数字，后面验证要用。

### 2.3 在源码中写出三个 XPath

三个 XPath 都以容器锚点为基准，用 `//`（后代轴）连接容器和内部元素：

#### list_item —— 列表项容器本身

```
//容器标签[@属性='值']
```

#### url —— 条目详情页链接，末尾带 `/@href`

从容器锚点出发，用 `//` 找到内部的 `<a>` 标签，取 `href` 属性：

```
//容器标签[@属性='值']//a[@class='链接class']/@href
```

如果 `<a>` 没有稳定 class，而包裹它的某个标签（如 `<h2>`）有稳定特征，则经过该标签：

```
//容器标签[@属性='值']//标题标签/a/@href
```

#### title —— 条目标题文本

定位到包含标题文字的标签本身。大多数情况下标题就在链接 `<a>` 标签内，此时 title 和 url 定位同一个 `<a>`（title 取文本、url 取 `@href`）：

```
//容器标签[@属性='值']//a[@class='链接class']/text()
```

如果标题在独立的 `<h2>` 等标签内（且不在 `<a>` 内），则定位到该标签：

```
//容器标签[@属性='值']//标题标签/text()
```

### 2.4 在源码中预验证（这一步在源码中完成，不动页面 DOM）

用三个 XPath 的思路，在源码文本中手动验证：

- **list_item**：源码中这组容器重复的次数，和页面上肉眼看到的条目数大致一致吗？
- **url**：每个容器内，按 url 的路径能找到 `<a href="...">` 且指向详情页（不是 `#`、不是 `javascript:`）吗？
- **title**：每个容器内，按 title 的路径能找到可读的标题文字吗？文字内容和页面上显示的一致吗？
- **误命中检查**：源码中其他区域有没有同标签名但结构不同的元素会被这三个 XPath 误命中？

不通过 → 回到 2.2，在源码中调整容器锚点。

---

## 第三步：在真实页面 DOM 中验证（仅验证，不修改）

**到了这一步，才第一次在真实页面上执行 XPath。** 目的是确认源码分析的结果在动态渲染后仍然正确。

执行三个 XPath，检查：

1. **命中数**：三个 XPath 命中数应一致，且与页面上主列表可见条目数大致匹配。
2. **归属正确**：抽查 2~3 组，每个 `url` 和 `title` 确实归属于对应的 `list_item` 容器。
3. **值正确**：`url` 提取到的链接指向详情页；`title` 提取到的文字与页面显示一致、无截断。
4. **区域纯净**：命中结果全部在主体内容区，无误入侧边栏、导航、横滑挂件、页脚。

验证不通过 → 回到第二步，在源码中重新分析。

---

## 第四步：分页检测

1. 在页面中找"下一页""加载更多""Next""More"等按钮 → 有则记录按钮 XPath，结束。
2. 没有按钮 → 滑动到页面底部 1 次，等约 1 秒。
3. 出现新列表项 → 记 `"scroll"`，否则记 `null`。

只滑 1 次。

---

## 第五步：输出

```json
{
  "list_item": "XPath|null",
  "url": "XPath|null",
  "title": "XPath|null",
  "next_page": "XPath|scroll|null"
}
```

输出 JSON 后任务完成。

---

# 第六步：分页数判定

在生成 DSL 之前，需要判定列表翻多少页。

## 核心规则

**以 T-1（昨天）到 T-2（前天）这个时间段内的列表项所经过的分页数，作为列表分几页的依据。**

- T = 当天
- T-1 = 昨天
- T-2 = 前天
- 时间窗口 = [T-2, T-1]，即前天和昨天这两天

## 判定流程

### 6.1 日期提取：在列表项容器内定位日期元素

**前提**：第五步已完成，`list_item` 容器的 XPath 已确定。以下所有日期提取操作都在单个 `list_item` 容器内进行。

#### 6.1.1 在容器内搜索日期 XPath

按以下优先级依次搜索，找到日期元素：

**优先级 1：`<time>` 标签**

`<time>` 是 HTML5 语义标签，专用于标记时间。通常带有 `datetime` 属性（机器可读的 ISO 格式）和文本内容（显示给用户）：

```
.//time/@datetime   →  "2026-05-11T08:30:00+08:00"
.//time/text()       →  "2026-05-11"
```

如果 `datetime` 属性存在，优先使用（格式规范、无需额外解析）。

**优先级 2：含日期关键词的 class/属性**

在容器源码中搜索以下关键词，定位日期所在标签：

- 英文：`date`, `time`, `publish`, `post`, `create`, `update`, `timestamp`, `when`, `posted`
- 中文：`时间`, `日期`, `发布`, `更新`
- 日文：`日付`, `投稿日`, `更新日`, `公開`
- 韩文：`날짜`, `시간`, `등록일`, `작성일`

示例：

```
.//span[contains(@class, 'date')]/text()
.//div[contains(@class, 'publish-time')]/text()
.//span[@class='post-meta-time']/text()
```

**优先级 3：常见日期容器结构**

有些站点用无 class 的 `<span>` / `<div>` 放日期，但位置固定（如标题下方第一个元素）：

```
.//div[@class='item-meta']/span[1]/text()
.//li//p[last()]/text()
```

**优先级 4：链接 URL 中的日期**

很多 CMS 的 URL 路径包含日期，如 `/2026/05/11/title-slug/`。从 `url` XPath 提取后用正则解析：

```
regex: /(\d{4})/(\d{2})/(\d{2})/
regex: /(\d{4})-(\d{2})-(\d{2})/
regex: /(\d{4})(\d{2})(\d{2})/
```

#### 6.1.2 日期格式识别与解析

拿到日期文本后，按以下常见格式尝试解析。**必须自己分析识别**，不要假设固定格式。

**数字日期格式：**

| 格式 | 示例 | 正则 |
|------|------|------|
| ISO 标准 | `2026-05-11` | `\d{4}-\d{2}-\d{2}` |
| 斜杠分隔 | `2026/05/11`, `05/11/2026`, `11/05/2026` | `\d{2,4}[/-]\d{2}[/-]\d{2,4}` |
| 点分隔 | `2026.05.11`, `11.05.2026` | `\d{2,4}\.\d{2}\.\d{2,4}` |
| 紧凑格式 | `20260511` | `\d{8}` |
| 中文日期 | `2026年5月11日` | `(\d{4})年(\d{1,2})月(\d{1,2})日` |
| 日文日期 | `2026年5月11日` | 同中文 |
| 韩文日期 | `2026년 5월 11일` | `(\d{4})년\s*(\d{1,2})월\s*(\d{1,2})일` |
| 英文月名 | `May 11, 2026`, `11 May 2026` | 见下文英文月份解析 |
| 带时间戳 | `2026-05-11 08:30`, `May 11, 2026 8:30 AM` | 先提取日期部分再解析 |

**相对日期格式（需换算为绝对日期）：**

| 格式 | 含义 | 换算 |
|------|------|------|
| `X 小时前` / `X hours ago` | X 小时前 | 今天 |
| `昨天` / `Yesterday` | 昨天 | T-1 |
| `前天` / `2 days ago` | 前天 | T-2 |
| `X 天前` / `X days ago` | X 天前 | T-X |
| `今天` / `Today` | 今天 | T |
| `刚刚` / `Just now` | 刚刚 | 今天 |
| `1 周前` / `1 week ago` | 一周前 | T-7 |

**英文月份缩写/全称对照：**

| 缩写 | 全称 | 月份 |
|------|------|------|
| Jan | January | 1 |
| Feb | February | 2 |
| Mar | March | 3 |
| Apr | April | 4 |
| May | May | 5 |
| Jun | June | 6 |
| Jul | July | 7 |
| Aug | August | 8 |
| Sep/Sept | September | 9 |
| Oct | October | 10 |
| Nov | November | 11 |
| Dec | December | 12 |

**多语言星期/相对词识别（用于确认而非解析）：**

- 中文：星期一~日、周一~日、今天、昨天、前天、刚刚
- 英文：Monday~Sunday、Today、Yesterday
- 日文：月曜日~日曜日、今日、昨日、一昨日
- 韩文：월요일~일요일、오늘、어제、그제

#### 6.1.3 解析流程

对每条列表项：

1. 在容器内执行日期 XPath，获取日期文本
2. 清理文本：去除前后空白、多余空格、`&nbsp;`
3. 尝试所有格式的正则匹配，提取年/月/日
4. 对于相对日期，以当前日期（T）为基准换算
5. 记录每条列表项解析后的日期（`datetime` 对象或 `(Y, M, D)` 元组）

**解析示例：**

```
"2026年5月11日"        → (2026, 5, 11)
"05/11/2026"           → (2026, 5, 11)  -- 注意美式月/日/年
"May 11, 2026"         → (2026, 5, 11)
"3小时前" (今天)       → T
"昨天"                 → T-1
"2026-05-11T08:30:00Z" → (2026, 5, 11)
"/2026/05/11/title"    → (2026, 5, 11)  -- URL 路径中提取
```

#### 6.1.4 日期提取失败时的渐进式兜底策略

如果按照 6.1.1 的四种优先级都无法提取到日期文本，按以下步骤逐级尝试：

**步骤 A：扩大搜索范围到容器父级**

有些站点日期不在容器内，而在容器紧邻的兄弟元素中。尝试在容器的父级下搜索：

```
// 假设 list_item 是 //div[@class='item']
// 尝试在父级下搜索（使用 ../.. 或直接在源码中观察父级结构）
```

**步骤 B：检查是否日期在列表以外的独立区域**

某些 SP 维护的站点日期不在每个容器内，而是按日期分组的（如 `<h3>2026年5月11日</h3>` 后面跟若干个该日期的条目）。这种情况下需要写额外的 XPath 来捕获分组日期。

**步骤 C：检查 JSON-LD 或 meta 标签**

```
//script[@type='application/ld+json']   →  JSON 中可能含 datePublished
//meta[@property='article:published_time']/@content
```

**步骤 D：完全无法提取日期**

只有在以上所有步骤均失败后，才使用兜底策略：

- **默认分 3 页**（`control.loop` stop=3），使用 3.4 模板（分页列表详情抓取）

### 6.2 判断 [T-2, T-1] 的条目是否全在首页

逐条检查首页列表项的解析后日期：

- 如果属于 [T-2, T-1] 的条目**全部在第 1 页** → **不分页**，使用 3.3 节模板（不分页列表详情抓取）
- 如果有属于 [T-2, T-1] 的条目**不在第 1 页** → 继续翻页查找，使用 3.4 节模板（分页列表详情抓取）

### 6.3 翻页查找，确定分页数

1. 点击"下一页"，到达第 N 页（N 从 2 开始）
2. 提取当前页所有列表项的日期
3. 检查当前页是否还有属于 [T-2, T-1] 的条目
4. 重复翻页，直到**当前页不再包含任何属于 [T-2, T-1] 的条目**
5. **stop = 最后一页的页码**（即 `control.loop` 的 `stop` 参数值）

### 6.4 判定示例

| 场景 | 判定结果 |
|------|----------|
| 属于 [T-2, T-1] 的条目全在第 1 页 | 不分页，使用 3.3 模板（不分页列表详情抓取） |
| T-1 条目在第 1 页，T-2 条目跨第 1~2 页 | 分页，stop=2，使用 3.4 模板（分页列表详情抓取） |
| T-1 条目在第 1 页，T-2 条目跨第 1~3 页 | 分页，stop=3，使用 3.4 模板（分页列表详情抓取） |
| 日期无法识别（已走完 6.1.4 全部兜底步骤） | 分页，stop=3，使用 3.4 模板（分页列表详情抓取） |

# 第七步：选择模板并生成 DSL 规则

## 7.1 读取参考文档（强制）

生成 DSL 前，必须读取 `references/操作组件规则配置文档.md`，理解各组件的参数规则。

**参考文档中的四套模板：**

| 章节 | 模板名称 | 适用场景 |
|------|----------|----------|
| 3.1 | 不分页列表抓取 | 仅抓列表 url/title，不打开详情页，无分页 |
| 3.2 | 分页列表抓取 | 仅抓列表 url/title，不打开详情页，有分页 |
| **3.3** | **不分页列表详情抓取** | **抓列表 + 打开详情页提取 html_content，无分页** |
| **3.4** | **分页列表详情抓取** | **抓列表 + 打开详情页提取 html_content，有分页** |

**模板选择铁律：优先选择列表详情抓取模版（3.3 / 3.4）。** 也即默认打开详情页提取完整 HTML 内容。仅当用户明确要求只抓列表页字段时才选择 3.1 / 3.2。

**关键组件速查（含详情页抓取所需组件的补充）：**

| 组件 | 用途 | 关键参数 |
|------|------|----------|
| `browser.open` | 打开列表页 | `url` |
| `data.select` | 选中列表项容器 | `selector`, `save_as`, `mode` |
| `control.loop` | 翻页循环 | `start`, `stop`, `step` |
| `control.foreach` | 遍历每个列表项 | `source`, `loop_item`, `item` |
| `data.enrich` | 注入元数据 | `target`, `fields` |
| `data.extract` | 提取字段值 | `source`, `target`, `fields` |
| `data.guard` | 数据校验 | `source`, `mode`, `save_as` |
| `control.if` | 条件判断 | `source`, `condition`, `target` |
| `data.push` | 推送下游 | `source`, `channel` |
| `element.click` | 点击跳转/翻页 | `source`, `selector`, `open_mode` |
| `browser.close_tab` | 关闭详情页标签 | — |
| `browser.switch_tab` | 切回列表页主标签 | `mode` |

## 7.2 模板选择决策树

按以下顺序依次判断，每一步排除不可能的选项，最终锁定唯一模板。

```
┌─ 决策 0：抓取模式（最高优先级，贯穿全程）
│   问：用户是否只要求抓列表页字段，不要求进入详情页？
│   ├─ 是（极少数情况）→ 使用列表抓取模板（3.1 或 3.2）
│   └─ 否（默认）      → 使用列表详情抓取模板（3.3 或 3.4）
│
├─ 决策 1：是否需要分页？（依据第六步判定结果）
│   问：数据是否需要翻页才能覆盖 [T-2, T-1]？
│   ├─ 不需要分页  → 模板锁定为 3.1（列表抓取）或 3.3（列表详情抓取）
│   └─ 需要分页    → 模板锁定为 3.2（列表抓取）或 3.4（列表详情抓取）
│
└─ 决策 2（仅分页）：分页触发方式？（依据第五步 next_page 输出）
    问：next_page 的值是什么？
    ├─ XPath 字符串  → 3.4.1 点击下一页按钮分页（见 7.3.1）
    ├─ "scroll"      → 3.4.2 滑动/点击加载更多分页（见 7.3.2）
    └─ null          → 实际上无分页，回退到 3.3
```

**决策示例：**

| 六步判定 | 用户要求 | 最终模板 |
|----------|----------|----------|
| 不分页 | 默认（无特殊要求） | **3.3** 不分页列表详情抓取 |
| 分页 stop=3 | 默认（无特殊要求） | **3.4** 分页列表详情抓取 |
| 不分页 | 用户明确说"只抓标题和链接" | 3.1 不分页列表抓取 |
| 分页 stop=2 | 用户明确说"只抓标题和链接" | 3.2 分页列表抓取 |

## 7.3 四套模板的组件流水线

### 7.3.1 模板 3.3：不分页列表详情抓取（最常用）

```
browser.open                   → 打开列表页
  data.select                  → 选中所有列表项容器
    control.foreach            → 遍历每个容器
      data.enrich              → 注入 meta 字段（lang, spider_source_id）
      data.extract (source=row) → 从容器提取 url / title 等列表字段
      data.guard               → 校验 url 和 title 非空
      control.if (guard_pass)  → 校验通过则进入详情页抓取
        element.click          → 点击列表项，open_mode="new_tab" 打开详情页
        data.extract (source=page) → 从详情页提取 html_content
        data.push              → 推送 kafka
        browser.close_tab      → 关闭详情页标签
        browser.switch_tab     → 切回列表页主标签，继续遍历
```

### 7.3.2 模板 3.4：分页列表详情抓取

**3.4.1 点击下一页按钮分页（next_page = XPath）：**

```
browser.open                   → 打开列表页
  control.loop                 → 翻页循环
    data.select                → 选中当前页所有列表项容器
    control.foreach            → 遍历每个容器
      data.enrich              → 注入 meta 字段
      data.extract (source=row) → 从容器提取 url / title
      data.guard               → 校验
      control.if (guard_pass)  → 通过则进入详情页
        element.click          → 打开详情页（new_tab）
        data.extract (source=page) → 提取 html_content
        data.push              → 推送 kafka
        browser.close_tab      → 关闭详情页
        browser.switch_tab     → 回到列表页
    element.click              → 点击"下一页"按钮
```

**3.4.2 滚动/加载更多分页（next_page = "scroll"）：**

```
browser.open                   → 打开列表页
  control.loop                 → 翻页循环
    element.click              → 点击"加载更多"或 browser.scroll 滑动到底部
    （循环结束后统一处理）
  data.select                  → 选中所有页的列表项容器
    control.foreach            → 遍历每个容器
      ...（同上：enrich → extract → guard → if → detail page → push）
```

> **scroll 模式与 click 模式的关键区别**：scroll 模式的分页操作在 `control.loop` 内完成（每次循环触发一次加载），而 `data.select` 在循环结束后执行，一次性选中所有已加载的列表项。

## 7.4 填充字段参数

### 7.4.1 data.select

| 参数 | 值 | 说明 |
|------|-----|------|
| `save_as` | `"list_rows"` | 固定值，供 foreach 引用 |
| `selector` | 第五步 `list_item` 加 `xpath=` 前缀 | `"xpath=//div[@class='item']"` |
| `mode` | `"cover"` | 每页覆盖（分页场景每次循环重新选择当前页） |

### 7.4.2 第一段 data.extract（列表页字段，source="row"）

在 foreach 内从列表容器提取字段。XPath 使用**相对路径**（以 `./` 或 `.//` 开头，相对于 select 选中的容器）：

**当 select 选中的是容器（如 `<div class='item'>`）时：**

| 字段 | 值 | 说明 |
|------|-----|------|
| `url` | `"xpath=.//h2/a/@href"` | 容器内标题链接的 href |
| `title` | `"xpath=.//h2/a/text()"` | 容器内标题链接的文本 |
| `link_url` | 同 `url` | 部分下游系统要求的冗余字段 |
| `link` | 同 `url` | 部分下游系统要求的冗余字段 |
| `new_no_content_list_title` | 同 `title` | 部分下游系统要求的冗余字段 |

**当 select 直接选中 `<a>` 标签时：**

| 字段 | 值 |
|------|-----|
| `url` | `"xpath=./@href"` |
| `title` | `"xpath=./text()"` |

**按需选填**（如果列表页有更多字段且用户需要）：

| 字段 | 典型 XPath | 说明 |
|------|-----------|------|
| `date` | `"xpath=.//time/text()"` | 发布日期 |
| `summary` | `"xpath=.//p[@class='desc']/text()"` | 摘要文字 |
| `author` | `"xpath=.//span[@class='author']/text()"` | 作者/来源 |

### 7.4.3 第二段 data.extract（详情页字段，source="page"）

进入详情页后，从整个页面提取字段。XPath 使用**绝对路径**：

| 字段 | 值 | 说明 |
|------|-----|------|
| `html_content` | `"xpath=//html"` | 全页 HTML，必填 |
| `content` | 视页面结构而定 | 正文区域文本，按需选填 |

如果详情页需要清理 DOM（去广告、去导航等），在 `html_content` 字段上加 processor：

```json
"html_content": {
  "selector": "xpath=//html",
  "processors": [
    { "type": "dom_clean" }
  ]
}
```

### 7.4.4 element.click（打开详情页，在 foreach 内）

| 参数 | 值 | 说明 |
|------|-----|------|
| `source` | `"row"` | 当前 foreach 循环项 |
| `open_mode` | `"new_tab"` | 在新标签页打开详情 |

> 如果 list_item 选中的就是 `<a>` 标签，则 `element.click` 直接点 source 即可，无需额外 `selector`。
> 如果 list_item 选中的是容器，且容器内 `<a>` 标签有明确的 XPath，则加 `selector` 参数指向该 `<a>`。

### 7.4.5 element.click（翻页按钮，在 loop 内、foreach 外）

仅分页模板需要。使用第五步输出的 `next_page` XPath，去掉末尾的 `/@href`：

```json
{
  "selector": "xpath=//a[@class='next page-numbers']"
}
```

如果分页方式为 scroll，则用 `browser.scroll` 或点击"加载更多"按钮替代：

```json
// 滑动到底部触发加载
{ "op_type": "browser.scroll", "params": { "mode": "bottom" } }

// 或点击加载更多按钮
{ "op_type": "element.click", "params": { "selector": "xpath=//button[@class='load-more']" } }
```

## 7.5 输出

生成完整的 DSL JSON。注意以下规则：

1. 每个 `children` 内的组件 `op_sequence` 从 1 重新编号
2. `spider_source_id` 使用用户提供的值，无则留占位符
3. `browser.open` 的 `url` 使用用户提供的列表页 URL
4. `control.loop` 的 `step=1`，`start=0`，`stop` 为第六步判定的分页数
5. 所有 selector 必须以 `xpath=` 开头

输出 JSON 后任务完成。

## 选择器规则

- 只用 XPath
- 每个标签名、class 名、属性值必须来自源码中真实出现过的内容
- 容器到内部元素用 `//`（后代轴），不要用 `/`（子节点）
- 列表页提取用相对路径（`./` 或 `.//` 开头），详情页提取用绝对路径（`//` 开头）
- `url` 末尾带 `/@href`

**严禁以下写法：**

- `contains(@href, '...')`
- `not(contains(...))` 堆叠
- `string-length()`、`normalize-space()`
- `[条件1 and 条件2 and 条件3]` 串联
